3-4.HTTPはシンプルなプロトコル

3-4.HTTPはシンプルなプロトコル

HTTPを用いた通信の特徴

  • ステートレス性
  • 最小限のメソッド

【ステートレス性】

　→通信時にサーバー側でクライアントの状態を保持しない。

　（⇔ステートフル）

ステートレスであることのメリット

　→サーバー側の負担が少ない。

　　サーバーの計算リソースを効率的に使用できる。


ステートレスであることのデメリット

　→クライアント側のパフォーマンス低下。

　　通信の度にログインなどでDBアクセスを繰り返すと、結果的にサーバーへの負担が大きくなる。

　　※クッキーやセッション管理による回避策

【最小限のメソッド】

　→HTTPに用意されているメソッドは8個のみ。

メソッド 処理内容
GET リソースの取得
POST リソースの作成・データ追加など
PUT リソースの更新・作成
DELETE リソースの削除
HEAD リソースのヘッダ情報取得
OPTIONS リソースのサポートメソッド取得
TRACE 自分宛にHTTPリクエストを返す
CONNECT プロキシ動作のトンネル接続への変更

4-1.RESTの概要

4-1.RESTの概要

【REST（Representational State Transfer）】

　→Webのネットワークシステムを構成する、

　　アーキテクチャスタイル。

　　分散システムに適した、リソースのやり取りに

　　関する設計原則の集合体。


　※アーキテクチャスタイル

　　→実際のシステムの具体的な設計（アーキテクチャ）に対する、指針や原則のこと。

RESTにおける4つの原則

  • ステートレスなクライアント／サーバ
  • リソースへの統一的なインタフェース
  • リソースの接続性
  • リソースのアドレス可能性

→WebはRESTfulなシステム。

4-2.RESTにおけるアーキテクチャスタイルの構成

4-2.RESTにおけるアーキテクチャスタイルの構成

RESTは複数のアーキテクチャスタイルで構成されている。

　→設計時には、必要なものを取捨選択することになる。


※以下、アーキテクチャスタイルの概要と特徴

【クライアント／サーバー】

　→クライアントとサーバーで処理を分離させる。

＜特徴＞

  • クライアント側をマルチプラットフォームにできる。
  • サーバー側はデータストレージとしての機能を提供するだけでよい。
  • 複数サーバーを組み合わせることで、可用性を向上させられる。

【ステートレスサーバ】

　→サーバー側で個別のクライアントの状態を記憶しない。

　　クライアントは1回の通信ごとに必要な全ての情報を送信する。

＜特徴＞

  • サーバー側の実装を簡略化できる。
  • リクエストに応えた後は、すぐに計算リソースを解放できる。

【キャッシュ】

　→サーバーから取得した情報をクライアント側に蓄積しておき、同じリソースにアクセスする際に再利用する。

＜特徴＞

  • クライアント／サーバー間の通信を減らすことができる。
  • 古いキャッシュが残っていると、予期しない動作をしてしまう。

【統一インタフェース】

　→リソースに対する操作を、統一した限定的なインタフェースでのみ行う。

＜特徴＞

  • インタフェースを限定する（制約する）ことで、全体の構成がシンプルになる。
  • 通信先の具体的な実装に影響されにくい。

【階層化システム】

　→システムを階層化させる。

＜特徴＞

  • クライアントやサーバーの接続先を意識せずに実装できる。
  • 古いシステムへの接続でも、間にWebサーバーを挟むことで、通信可能となる。

【コードオンデマンド】

　→コードをクライアント側で実行する。

＜特徴＞

  • サーバーと通信することなく、クライアント側の処理内容を増やせる。
  • 処理の実行速度が上がる。
  • サーバー側とやり取りしているリソースの内容が不明瞭になる。
